# 高校宿舍维修报修管理系统概要设计说明书

## 一、项目概述

### 1.1 项目名称

基于 Spring Boot 的高校宿舍维修报修管理系统设计与实现

### 1.2 建设背景

高校宿舍日常维修通常依赖电话、纸质登记或人工转述，存在登记分散、处理进度不透明、责任追踪困难、统计分析不便等问题。为提升宿舍维修业务的信息化水平，本系统围绕“学生报修、管理员审核派单、维修人员处理、学生评价反馈”的主流程，设计并实现一套前后端分离的宿舍维修报修管理系统。

### 1.3 建设目标

本系统的目标如下：

1. 实现宿舍维修业务线上化，降低人工登记成本。
2. 实现报修工单全流程流转，便于查看处理进度。
3. 实现角色分工管理，支持学生、维修人员、管理员三类用户协同使用。
4. 实现公告发布、评价记录、统计分析等辅助功能，为宿舍管理提供数据支持。

### 1.4 设计范围

本次概要设计以当前已完成的核心业务功能为范围，重点覆盖以下内容：

1. 登录注册与身份识别
2. 报修工单管理主流程
3. 宿舍楼、宿舍房间、报修类型等基础数据管理
4. 维修评价、公告管理与统计分析

部分扩展能力如操作日志页面、个人资料维护等，数据库已有预留或结构可扩展，但不作为本版核心实现范围。

## 二、总体设计

### 2.1 系统角色

系统包含三类业务角色：

1. 学生：提交报修、查看工单、取消待审核工单、对已完成工单进行评价。
2. 维修人员：查看分配给自己的工单，开始维修、完成维修，查看历史工单和个人评价。
3. 管理员：管理用户、宿舍楼、宿舍房间、报修类型、工单审核与派单、公告发布及统计分析。

### 2.2 总体架构

系统采用前后端分离架构，整体分为前端表示层、后端业务层和数据库存储层。

```text
+-------------------------------------------------------------+
|                      前端表示层（Vue 3）                     |
|  学生端页面     维修人员端页面     管理员端页面              |
+-----------------------------+-------------------------------+
                              |
                              | HTTP + JSON + JWT
                              v
+-------------------------------------------------------------+
|                 后端业务层（Spring Boot 3）                 |
| Controller  Service  Mapper  Interceptor  Config            |
+-----------------------------+-------------------------------+
                              |
                              | MyBatis-Plus / JDBC
                              v
+-------------------------------------------------------------+
|                      数据存储层（MySQL）                     |
| 用户表  工单表  评价表  公告表  维修记录表  基础数据表等     |
+-------------------------------------------------------------+
```

### 2.4 后端分层设计

后端工程采用典型分层结构：

1. `controller` 层：对外提供 RESTful 接口。
2. `service` 层：封装业务逻辑与流程控制。
3. `mapper` 层：负责数据库访问。
4. `entity` 层：定义数据库表对应的实体对象。
5. `interceptor` 层：负责登录拦截与 JWT 校验。
6. `config` 层：负责跨域、拦截器、静态资源映射等配置。
7. `common` 层：提供统一返回结果、异常、工具类等公共能力。

## 三、功能模块设计

### 3.1 登录认证模块

该模块负责用户注册、登录、获取当前登录用户信息及修改密码。

主要设计如下：

1. 用户注册时写入用户基础信息，密码采用 BCrypt 加密后存储。
2. 用户登录成功后由后端生成 JWT，前端保存令牌并在后续请求中携带。
3. 后端通过登录拦截器校验令牌合法性，并将 `userId`、`username`、`role` 写入请求上下文。
4. 前端根据用户角色进入学生端、维修人员端或管理员端页面。

### 3.2 学生端模块

学生端主要完成报修申请和结果反馈，包含以下功能：

1. 提交报修工单
2. 上传报修图片
3. 查看我的报修列表
4. 查看工单详情与维修记录
5. 取消待审核工单
6. 修改被驳回工单后重新提交
7. 对已完成工单进行维修评价

### 3.3 维修人员端模块

维修人员端围绕执行工单展开，包含以下功能：

1. 查看待处理工单
2. 查看工单详情
3. 开始维修
4. 提交维修结果并完成工单
5. 查看历史维修工单
6. 查看本人获得的评价

### 3.4 管理员端模块

管理员端承担系统运营与业务调度职责，包含以下功能：

1. 用户管理
2. 宿舍楼管理
3. 宿舍房间管理
4. 报修类型管理
5. 报修工单管理
6. 工单审核
7. 工单派单
8. 公告管理
9. 评价查看
10. 数据统计分析

### 3.5 公共辅助模块

公共辅助模块主要包括：

1. 图片上传与访问
2. 统一返回结果封装
3. 分页查询与条件筛选
4. 维修记录跟踪
5. 统计报表展示

## 四、业务流程设计

### 4.1 报修业务主流程

报修工单的核心流转过程如下：

```text
学生提交报修
   |
   v
[0] 待审核
   | \
   |  \ 学生取消
   |   \
   |   [7] 已取消
   |
   +--> 管理员审核不通过 --> [1] 审核驳回
   |                           |
   |                           +--> 学生修改后重新提交 --> [0] 待审核
   |
   +--> 管理员审核通过 --> [2] 待派单
                              |
                              +--> 管理员派单 --> [3] 待维修
                                                    |
                                                    +--> 维修人员开始维修 --> [4] 维修中
                                                                                 |
                                                                                 +--> 完成维修 --> [5] 已完成
                                                                                                      |
                                                                                                      +--> 学生评价 --> [6] 已评价
```

### 4.2 状态设计

当前系统中报修工单状态定义如下：

| 状态值 | 状态名称 | 说明 |
|--------|----------|------|
| 0 | 待审核 | 学生已提交，等待管理员审核 |
| 1 | 审核驳回 | 管理员审核未通过，允许学生修改后重新提交 |
| 2 | 待派单 | 审核通过，等待管理员分配维修人员 |
| 3 | 待维修 | 已派单，等待维修人员处理 |
| 4 | 维修中 | 维修人员已开始处理 |
| 5 | 已完成 | 维修人员已完成维修 |
| 6 | 已评价 | 学生已完成评价 |
| 7 | 已取消 | 学生取消工单 |

### 4.3 维修记录设计

系统通过维修记录表记录关键业务操作，用于展示工单处理过程。典型记录包括：

1. 提交报修
2. 审核通过
3. 审核驳回
4. 派单
5. 开始维修
6. 完成维修
7. 取消工单

## 六、接口设计

### 6.1 设计原则

系统接口采用 RESTful 风格，前后端通过 JSON 数据交互，统一返回格式包含 `code`、`message`、`data` 三个字段。

### 6.2 核心接口模块

#### 1. 认证接口

1. `POST /api/auth/register`：用户注册
2. `POST /api/auth/login`：用户登录
3. `GET /api/auth/info`：获取当前登录用户信息
4. `PUT /api/auth/password`：修改密码

#### 2. 工单接口

1. `POST /api/repair-orders`：提交报修
2. `PUT /api/repair-orders/resubmit/{id}`：驳回后重新提交
3. `GET /api/repair-orders/page`：分页查询工单
4. `GET /api/repair-orders/{id}`：查询工单详情
5. `PUT /api/repair-orders/cancel/{id}`：取消工单
6. `PUT /api/repair-orders/audit/{id}`：管理员审核
7. `PUT /api/repair-orders/assign/{id}`：管理员派单
8. `PUT /api/repair-orders/start/{id}`：开始维修
9. `PUT /api/repair-orders/finish/{id}`：完成维修

#### 3. 评价接口

1. `POST /api/evaluations`：学生提交评价
2. `GET /api/evaluations/order/{orderId}`：查询工单评价
3. `GET /api/evaluations/page`：管理员分页查询评价
4. `GET /api/evaluations/my`：维修人员查看本人评价

#### 4. 基础数据接口

1. 用户管理接口：`/api/users`
2. 宿舍楼管理接口：`/api/buildings`
3. 宿舍房间管理接口：`/api/rooms`
4. 报修类型管理接口：`/api/repair-types`
5. 公告管理接口：`/api/notices`

#### 5. 统计接口

1. `GET /api/statistics/overview`：统计概览
2. `GET /api/statistics/monthly`：月度报修统计
3. `GET /api/statistics/type`：报修类型占比
4. `GET /api/statistics/repair-user`：维修人员完成数量排行
5. `GET /api/statistics/satisfaction`：满意度统计

## 七、权限与安全设计

### 7.1 认证设计

系统采用基于 JWT 的无状态认证机制：

1. 用户登录成功后由后端生成 JWT。
2. 前端将令牌保存到本地，并在请求头中携带 `Authorization: Bearer <token>`。
3. 后端登录拦截器统一校验令牌合法性。
4. 校验通过后，将用户 ID、用户名、角色写入请求上下文，供后续业务接口使用。

### 7.2 权限设计

当前系统以角色隔离为主：

1. 学生只能操作自己的报修工单与评价数据。
2. 维修人员只能处理分配给自己的工单，并查看自己的评价。
3. 管理员负责审核、派单和基础数据维护。

说明：本系统当前以毕业设计功能实现为重点，权限控制已覆盖核心业务接口，后续仍可继续增强更细粒度的页面级与按钮级权限管理。

### 7.3 密码与资源安全

1. 用户密码采用 BCrypt 加密存储，不以明文落库。
2. 后端通过拦截器限制未登录访问。
3. 图片资源通过静态资源映射方式提供访问，上传目录与业务接口分离。

## 八、前端页面设计

### 8.1 页面结构

前端按角色划分页面模块：

1. 公共页面：登录页、注册页
2. 学生端页面：首页、提交报修、我的报修、工单详情、评价页
3. 维修人员端页面：首页、待维修工单、工单详情、历史记录
4. 管理员端页面：首页、数据统计、用户管理、宿舍楼管理、房间管理、报修类型管理、工单管理、公告管理、评价查看

### 8.2 页面设计思路

1. 采用左侧菜单加顶部信息区的后台管理布局。
2. 以表格、表单、对话框为核心交互方式，满足信息录入与列表管理需求。
3. 统计页面通过 ECharts 展示概览卡片、趋势图和占比图，提高数据可读性。

## 九、部署设计

### 9.1 开发部署结构

系统在开发环境中由三部分组成：

1. 前端项目 `dorm-repair-ui`
2. 后端项目 `dorm-repair`
3. MySQL 数据库

### 9.2 运行方式

1. 先导入数据库建表脚本与初始化数据。
2. 启动 Spring Boot 后端服务。
3. 启动 Vue 前端项目，通过浏览器访问系统。
4. 前端通过 HTTP 调用后端接口完成业务交互。
